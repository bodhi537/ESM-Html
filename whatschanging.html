What's changing?

  • IT Service Management
    • Incident Management

      Before

      Major Incident process handled outside BMC
      Option to ‘Convert INC to WO’
      All Linked Records added in the same Relationships Tab
      Incident Priority: Critical – High – Medium - Low

      After

      Major Incident Workbench inherent in ServiceNow
      Cannot convert INC to WO (Can be categorized as WO in ‘Resolution Code’ Field)
      Separate Fields and Tabs for Related Records (Parent/Child INCs, Problem, Caused by CRQ, Fix CRQ)
      New ‘Planning’ Priority for Low Impact + Low Urgency Incidents
    • Problem Management

      Before

      Unable to flag Problems with inconclusive Root Cause Analysis
      All investigation results entered in same Work Details Tab
      Configuration Item (CI) not required

      After

      ‘Accept Risk’ functionality (to identify Problems with no permanent resolution)
      “Sequence of Events” captured separately from “Analysis Info” (3 Steps: Workaround, Root Cause and Fix Notes)
      Configuration Item (CI) required (Same as for Incident Management)
    • Change Management

      Before

      5 Risk Categories
      Minimum 7 days lead time for normal changes
      Changes are classified in 4 classes (Normal, Expedited, Standard, Emergency)
      Emergency changes can be approved after implementation

      After

      3 Risk Categories (Low, Moderate, High)
      Flexible lead time for normal changes
      Changes are classified in 3 classes (Normal, Standard, Emergency)
      Emergency changes only used in very specific situations(Critical/high incidents, Implement ≤ 24 hrs, approvals prior to implementation*)
    • Forms and User Portal

      Before

      Forms are created by BMC centralized tools team
      Changes to IT Services Portal segregated from changes to services catalog
      One single portal and service catalog for all users
      BMC forms as front-end to create Work Orders assigned to specific queues

      After

      Catalog Items created and maintained by process teams
      Integrated portal & service offerings (once a new catalog item is created it's automatically available in Self-Service portal)
      Different Service Catalogs
      Service Requests replacing Work Orders
  • IT Operations Management
    • Reporting and Analytics

      Before

      All reports & dashboards custom
      Reports and dashboards built on ITDW
      Operational data from BMC sent to ITDW for consumption

      After

      Wealth of OOTB reports, dashboards and KPIs
      Reports and dashboards created directly in ServiceNow (for data sourced in ServiceNow)
      Operational data from ServiceNow sent to ITDW for consumption
    • Data including CMDB

      Before

      Super-crappy 1990’s CMDB interface
      Centralized discovery
      Lack of consistent data ownership

      After

      Easier to use, intuitive interface
      Federated discovery
      Comprehensive data governance
  • Software Asset Management
    • Software Asset Management

      Before

      7 ITSM forms (existing, new, complex, license, retire, server)
      Manual handoffs and work orders between teams
      Limited visibility into software costs

      After

      Enhanced “shopping cart” style experience
      Automated workflows and streamlined integration with other EMIT/non-EMIT groups
      Visibility into ownership, cost, compliance, and optimization opportunities
  • ServiceNow Platform
    • Platform Architecture & Support

      Before

      On-Premise
      Centralized support
      Highly customized

      After

      Cloud SaaS solution
      Self-Service & Support provided by MSP
      Utilizing Out Of The Box functionality
    • APIs

      Before

      Ticketing API on-promise
      Kerberos Authentication
      Simplified API

      After

      Customers will direct connect to ServiceNow API which will require port openings
      Customers API connection will need to utilize oAuth (basic authentication requires an exception)
      More robust API (brings more features and capabilities)
    • Notable System Integrations

      Before

      Waterfall Deployment
      Stored Credentials
      Centralized Automation

      After

      E-Bonding Strategy
      CyberArk Capability
      Re-useable Automation Offering
